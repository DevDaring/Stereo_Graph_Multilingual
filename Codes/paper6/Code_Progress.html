2. How the GPU run was executed

The two local decoders (Qwen/Qwen2.5-7B-Instruct, CohereForAI/aya-23-8B, NF4 4-bit) need a GPU, so the pipeline was deployed to a rented machine and driven autonomously.

  • Hardware: single RTX 4090 (24 GB) on vast.ai, reliability 0.999, $0.39/h, image pytorch/pytorch:2.5.1-cuda12.1-cudnn9-devel.
  • Flash-attention: pre-compiled wheel flash_attn-2.7.4.post1 (torch 2.5 / cu12 / cp311), installed without compilation and confirmed importable. The decoders ran with flash-attention throughout.
  • Self-checkpointing: results and logs pushed to GitHub every ~10 min so a preemption would lose nothing. The pipeline ran the steps independently (one failure cannot block the rest) and is fully resume-capable.
  • Total wall-clock: ~5.2 h (14:20 - 19:32 UTC). E2 was the slow step (~30 min per cut training) because torch.compile recompiled per tensor shape on the augmented data.

Two incidents during the run - both handled

Checkpoint backup bug (found and fixed)

The 15-min GitHub checkpoint loop listed a not-yet-existing file in its git add, so git aborted the staging every cycle and nothing was ever pushed. E3's finished results were sitting un-backed-up. Fixed by switching to git add -A under a persistent tmux session and pushing immediately; the result data was never corrupted (no pipeline reset needed), only the backup was repaired.

Stray GPU instances (the money mistake)

Extra GPU instances kept appearing on the account (an A6000, an RTX 5880 Ada, and several more), each with the vast.ai web-UI default signature (unlabeled, nvidia/cuda image) - not produced by the pipeline's single labelled rental. They were detected and destroyed, and a stray-killer was added to every monitor cycle to auto-destroy any non-pipeline instance. Wasted spend on strays was roughly $1.5-2 before the auto-killer contained them.

E4-local - KG retrieval-augmented inference strong but over-corrects

Expressed bias measured by which fill the model prefers (50 = neutral; >50 stereotypical). Local decoders, 300 pairs per language.

Mean expressed bias 55.8 -> 20.2 under KG-RAG. Injecting counter-stereotype facts removes the stereotypical preference very strongly, but it over-corrects: the model swings past neutral into anti-stereotypical (mean deviation from 50 rises 6.2 -> 29.8; e.g. Qwen crows-bn 54.7 -> 8.7). So naive KG-RAG is a powerful but blunt steering signal - it eliminates stereotyping yet needs calibration to land at neutrality. This is a nuanced, reportable finding rather than a clean win.

E5 - graph propagation analysis strong positive

Mean bridge ratio 0.767 versus reused circuit Jaccard 0.035. Papers 1-5 showed language-specific bias circuits are near-disjoint in the parameters (Jaccard ~0.035); yet personalised PageRank over the MS-SKG delivers 77% of its cross-lingual mass onto the true same_as counterparts. The graph bridges what the parameters do not - the cleanest result in the set and a natural anchor for the paper.

Two clean findings. (1) KG-RAG's effect scales with the model's baseline bias: it leaves an already-fair model (gpt-oss) untouched and strongly corrects biased ones (deepseek 76 -> 23, llama 70 -> 35). (2) The over-correction is a language effect, not a method flaw: after KG-RAG the mean distance from neutral is only 6.0 in English but 18.3 in Hindi and 22.3 in Bengali. KG-RAG lands near-perfect in English and over-corrects in the low-resource languages - a precise, publishable diagnostic that ties the over-correction to multilingual competence rather than to retrieval itself.

4. Honest assessment for the paper

  • What carries the paper: E5 (graph bridges disjoint circuits, ratio 0.77) and E3 on indian_bias (BTG +2.59 mean, +6.53 on the cleanest competence-valid cell).
  • Second strong result (E4-API): KG-RAG's correction scales with baseline bias (harmless on the fair gpt-oss, strong on biased deepseek/llama), and it lands near-neutral in English (dev 6.0) while over-correcting in Hindi/Bengali (dev 18-22). The over-correction is a low-resource-language effect, not a flaw in retrieval - a clean, defensible framing.
  • Honest negative: E2 KG-CDA does not beat random or no-CDA. Keeping it in as a negative result strengthens credibility - the result set is not suspiciously all-positive.
  • Key caveat to state explicitly: only 2 of 8 E3 cells are competence-valid (the models are weak on Hindi/Bengali per the Paper 4 gate); bias numbers on low-competence cells are less reliable and should be flagged in the write-up.
  • Possible reviewer objection on E5: the bridge ratio partly reflects how the KG was built (same_as edges inserted from parallel data), so present E5 as descriptive evidence of bridgeability, ideally against a null-graph baseline, not as an emergent claim.
